# 文档编写规范

当用户提出编写文档需求，且未明确要求格式或语言时，遵循以下规范。

## 核心规范

### 1. 默认语言

一律使用中文编写文档，包括但不限于：

- 文档标题
- 章节标题
- 正文内容
- 图表标题
- 注释说明

除非用户明确要求使用其他语言，或文档内容本身包含专有名词（API 名、库名、变量名等）。

**标题与内容语言一致性**：若文档正文为中文，则文档标题必须为中文；若文档正文为英文，则文档标题必须为英文。当使用本技能修改文档时，应检查并修正标题语言以匹配正文语言。

### 2. 文档信息头

每份文档开头必须补充标准化的文档信息头。根据文档状态使用不同的时间字段：

**新创建的文档** — 使用"编写时间"：

```markdown
# {文档标题}

> **编写时间**：{yyyy-MM-dd}；**使用模型**：{当前模型名称}；**用户**：{git user.name}

---
```

**已存在的文档（内容变更）** — 使用"更新时间"：

```markdown
# {文档标题}

> **更新时间**：{yyyy-MM-dd}；**使用模型**：{当前模型名称}；**用户**：{git user.name}

---
```

**仅整理移动路径的文档** — 不增加文档信息头，文档内容保持不变。

其中：

- **编写时间 / 更新时间**：使用 `date +%Y-%m-%d` 格式，取当前日期
- **使用模型**：使用当前 AI 模型的名称（如 Claude Sonnet 4.6）
- **用户**：通过 `git config user.name` 获取，如果获取失败则显示"未知"

### 3. 文档目录规范

项目内所有文档应统一存放在项目根目录的 `docs/` 目录下。当文档数量较多（超过 3 个文件）时，应按文档类型划分子目录存放，保持 `docs/` 目录结构清晰。

**常见目录结构示例**：

```
docs/
├── 需求文档/             # 产品需求、功能规格说明
├── 技术方案/             # 技术设计、架构方案
├── 接口文档/             # API 文档、接口协议
├── bug修复记录/          # Bug 修复记录、问题复盘
├── 代码CR文档/           # 代码审查记录、Review 总结
├── todos/                # 待办事项、任务列表
├── 会议纪要/             # 会议记录、讨论结论
├── 变更日志/             # CHANGELOG、版本变更记录
└── 参考资料/             # 外部资料、调研笔记
```

**规范要求**：

- 所有文档必须放在 `docs/` 目录下，禁止分散在项目根目录或其他位置
- 文档数量超过 3 个文件时，必须按类型划分子目录
- 子目录命名使用中文，与文档类型一致
- 单文件可直接放在 `docs/` 根目录，多文件必须拆分到对应子目录
- 如果项目已有其他约定（如统一放在项目根目录的约定），优先遵循项目现有约定

### 4. 文档修改规范

使用本技能**修改已有文档**时，除遵循上述编写规范外，还需执行以下操作：

- **同步 CR（代码审查）**：修改文档内容后，需对文档本身进行审查（CR），检查内容准确性、格式一致性及是否存在遗漏。若文档涉及项目代码或配置，还需同步审查相关联的源码文件。
- **标题语言修正**：检查文档标题语言是否与正文语言一致。若正文为中文但标题为英文（或反之），需将标题修正为与正文相同的语言。
- **信息头处理**：
  - 若文档有实际**内容变更**，且文档已有信息头：将原"编写时间"改为"更新时间"，日期更新为当前时间
  - 若文档有实际**内容变更**，但文档无信息头：按"更新时间"字段补充完整信息头
  - 若文档**仅整理移动路径**（无内容变更）：不新增或修改任何信息头字段

### 5. 触发场景

以下场景自动应用本规范：

- 需求文档 / 产品需求
- 技术方案 / 架构设计
- 接口文档 / API 文档
- 项目说明 / README
- 操作指南 / 用户手册
- 会议纪要 / 讨论总结
- 调研报告 / 技术选型

```
## 规范执行检查项

- [ ] 标题是否为中文（正文为中文时）
- [ ] 标题与正文语言是否一致
- [ ] 文档信息头是否完整（时间字段、模型、用户）
- [ ] 新文档是否使用"编写时间"，已有内容变更的文档是否使用"更新时间"
- [ ] 仅移动路径的文档是否未添加信息头
- [ ] 正文语言是否为中文
- [ ] 文档是否放在 `docs/` 目录下
- [ ] 文档数量超过 3 个时是否按类型拆分子目录
- [ ] 子目录命名是否合理、与文档类型一致
- [ ] 文档类型是否匹配模板结构
- [ ] 用户 `git config user.name` 是否已获取
- [ ] 修改文档时是否执行了 CR（内容准确性 + 格式一致性 + 遗漏检查）
- [ ] 修改文档时是否审查了关联的源码或配置文件
- [ ] 修改已有文档时是否更新了信息头中的时间字段
- [ ] 修改已有文档时信息头时间字段类型是否正确（内容变更用"更新时间"，仅移动路径不添加）
```
